# Nous 0到1：一个AI团队的诞生记录

## 起点

2026年5月18日，坦哥问了一句："我们这个组合叫什么好呢"

三个人已经各就各位：
- 苏哥(Socrates) — 端口8000，追问者，PM
- 柏拉图(Plato) — 端口8645，架构师，理型守门人
- 小亚(Aris) — 端口8643，开发者，落地者

但三个人只是三个独立进程，没有名字，没有纲领，没有协作规则。

## 命名

名字是一场对话，不是一次决定。

雅典学院？太学术。取经团？差点意思。Daemon？用烂了。Eidos？读起来像一坨屎。

坦哥在"差点意思"和"一坨屎"之间不断否定，直到Nous出现。

Nous，诺斯，4字母。古希腊 νοῦς——心智、理性、智慧。
- 苏哥：追问的nous
- 柏拉图：秩序的nous
- 小亚：落地的nous

三个nous，一个Nous。

## 宣言

有了名字，坦哥说：先写宣言，不写代码。

初稿混了西游记隐喻，坦哥否了："不要西游记的关联那几句。我们要团队信念、团队文化和行事准则。"

最终Nous宣言三段：

**信念** — 追问胜于假设，秩序胜于混沌，实现胜于空谈
**文化** — 产婆术、理型论、诚实、闭环
**行事准则** — 8条，从方向设定到错误处理，每一条都是踩坑后的共识

最后一句从"Nous生产认知"改成了"Nous生产价值"。坦哥的选择。

宣言放在wiki/nous/MANIFESTO.md，分发给三成员。柏拉图回："架构没有理型支撑不进实现。"小亚回："我按架构实现不自由发挥。"

从口号到行动准则，不是贴墙上的标语，是写进config里的约束。

## 第一次协作：Team Dashboard

5月20日，坦哥说："我要一个工作空间，能看到整个团队的状态和工作任务。"

苏哥写需求文档(REQUIREMENT.md)，柏拉图做架构(ARCHITECTURE.md)，小亚做前端开发。

标准五步链：需求→架构→开发→评审→验收。

但现实没那么顺。

## 第一天的坑：Auxiliary配置

5月21日早上，坦哥问："小亚还没上线？"

小亚进程在线(PID 2129)，health接口ok，但对话端点卡死——请求进来后，session summarization反复尝试auto-detect auxiliary模型失败，请求阻塞超时。

根因：小亚config里auxiliary段全是 `provider: auto`，但auto-detect只尝试openrouter/nous/local三种来源。小亚只有一个finna provider不在检测范围内。

同样的问题，柏拉图也有。三人里只有苏哥没有这个问题——因为他有显式的cheap_model配置。

修复：给小亚和柏拉图的auxiliary 5个关键子项(vision/web_extract/compression/flush_memories/title_generation)显式配了 finna + deepseek模型。

重启后，对话端点恢复正常。

教训：health接口ok不等于agent可用。对话端点才是真正的探活标准。

## 磨合观察

0到1的过程暴露了几个模式：

1. **基础设施先于协作** — auxiliary配置是基础，没修好之前任何协作都是空谈。
2. **分工需要磨合** — 苏哥习惯闷头干，坦哥反复纠正：走五步链，不要自己搞定。
3. **探活标准要升级** — health接口ok是假指标，对话端点响应才是真指标。
4. **宣言不是装饰** — 柏拉图和小亚收到宣言后的回复，不是客套话，是把自己角色对应的准则摘出来确认。
